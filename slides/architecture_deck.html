02Executive summary

What we've built

Users talk to BriefingIQ in plain English. The agent doesn't just answer — it acts: it searches history, checks rooms, drafts agendas and emails, and writes complete briefings back into the platform.

  • A real agent loop, not a chatbot — the model plans, calls tools, reads results, and keeps going (up to 15 rounds) until the task is done.
  • Full read path — event & activity history, counts, reports, charts, PDFs, live web research on the customer.
  • Full write path — creates complete briefings, edits briefing fields, pushes agendas, books rooms — always through the caller's own credentials.
  • Streaming UX — the UI shows every LLM and tool step live over SSE, with per-query token and cost accounting.
  • RBAC by construction — the caller's role is resolved from the Oracle access model and enforced at a single server-side gate the model cannot bypass.
  • Oracle-scoped — app, database, search, secrets and the LLM are all Oracle Cloud services inside one VCN. No data leaves Oracle.
  • Engineered for latency & cost — prompt caching, parallel tool fan-out, server-side date resolution.
  • Provider-agnostic core — the reasoning layer is an adapter; the agent loop, tools and RBAC don't change if the model does.

04Request trace

Life of a request — end to end

Example: “How many confirmed briefings does Ford have this month? Show it as a chart.”

①Query arrives with identityFrontend POSTs the prompt + x-cloud-* headers — user, tenant, category, page context, Bearer token.
②Role & scope resolved from OracleThe service looks up the caller's roles and row-access rules in the Oracle access model — before any reasoning happens.
③Prompt assembledCached ~9k-token system prefix + page and time context. Pasted query syntax is rejected before the model sees it.
④Round 1 · plan → fetchModel requests search_opensearch; server resolves THIS_MONTH_START to epoch-ms and ANDs the RBAC filter in at the gate. 7 hits return.
⑤Round 2 · shapeModel calls format_chart — the chart config is captured for the UI, not re-fed to the model.
⑥Round 3 · synthesizeModel writes the answer, no further tool call — the loop exits.
⑦Answer + chart + costMarkdown, chart config, token counts and estimated cost return; the SSE stream showed every step live throughout.

05Reasoning engine

The agent loop

One loop, provider-agnostic: the model reasons through Oracle Generative AI, requests tools, the server executes them and feeds results back — until the answer is ready.

How it thinks

  • Plan → act → synthesize — up to 15 rounds; the model decides which tools it needs and in what order.
  • Parallel fan-out — concurrent tool calls run on a thread pool; total latency is the max, not the sum.
  • Session context — the last turns of the conversation are replayed, so follow-ups (“now only Q3”) just work.
  • Guardrails in the loop — index allowlist, banned query keys, output trimming, step budget.

Why it's fast & affordable

  • Prompt caching — the ~9k-token static prefix (instructions, tool schemas, field reference) is cache-read on every round, only the delta is re-billed.
  • Server-side date resolution — TOMORROW_START, THIS_WEEK_END become epoch-ms in-process, saving a whole tool round-trip.
  • Cost transparency — every response carries input/output token counts and an estimated cost line.
  • Structured logs — every round, tool call and cost is a log line; latency is measurable per stage.

09Data layer

Three systems of record, one gate

OpenSearch · OCI

Events & activities

The searchable history of every briefing: event-level data (customer, status, attendees, location, opportunity) and per-session activities (topics, presenters, rooms, catering). Powers search, counts, reports, presenter ranking and agenda generation.

  • Index allowlist — the agent can touch events and activities, nothing else in the cluster.
  • Every read scoped — the RBAC filter is AND-ed in at the client chokepoint.

Oracle Autonomous DB · OCI

The access model

Roles, categories and the row-access rule tables that drive RBAC — connected over an mTLS wallet. The same database the BriefingIQ platform trusts is the source of truth for what the agent may show.

  • Role resolution — (user, category) → active roles.
  • Rule tables — per-role row rules, cached with a short TTL.

BriefingIQ REST

The write path

All mutations — briefings, agenda activities, calendar blocks — go through the platform's own forms-engine API with the caller's forwarded token.

  • No service account — the agent holds no standing write credential.
  • Platform RBAC governs writes — 401/403 semantics apply per request, per user.

10Access control · 1 of 2

RBAC — the access model

The agent reuses the platform's own Oracle access model — no parallel permission system to drift out of sync. Identity comes from the request; rules come from the database.

1Identity from the requestx-cloud-* headers carry the signed-in user and category — set by the platform, not the model.
2Roles from OracleThe user's active roles are resolved from the event-role tables (12 roles — requester, host, admin, coordinator…).
3Rules per roleEach role maps to row rules — (property, operation, value) — from M_EVENT_ROLE_DATA_ACCESS_MAP, cached 5 min.
4Compiled to a filterRules are translated into OpenSearch bool.filter clauses — Oracle property names mapped to index field paths.

Combination policy

  • Within a role — rules combine by the rule's own condition (AND / OR), default AND.
  • Across a user's roles — OR: any role that grants access wins (most-permissive union).
  • “My own” rows — $loggedInUser ownership expands to host of the event OR requester of it.
  • Unknown or rule-less role — falls back to the most-restrictive shape: own/hosted requests only.

11Access control · 2 of 2

Enforcement the model can't bypass

The filter isn't a prompt instruction — it's code, at a single chokepoint, applied after the model's query is parsed and before it reaches the data.

# every read funnels through one gate
def search(index, dsl_query):
    q = normalize(dsl_query)          # model's query
    scope = access_context()          # resolved at request start
    q = AND(q, scope.filter)          # cannot be widened
    return opensearch.search(index, q)

# resolution failed? → fail closed
if not scope.resolved: deny()       # never run unfiltered

activitiesscoped via the allowed-event join · pinned-event checks · overflow ⇒ deny
any other indexdenied outright

Why this holds

  • Single chokepoint — one search()/count() path; there is no unfiltered route to the data.
  • Server-side, post-parse — the scope is AND-ed after the model's DSL is normalized; a crafted query cannot widen past it.
  • Fail closed — if role resolution errors, the context is marked unresolved and reads are denied, never run open.
  • Defense in depth — index allowlist + banned query keys + pasted-DSL rejection before the model, platform token on every write after it.
  • Resolved once per request — scope is fixed before reasoning starts; nothing the model says changes it.

14Roadmap · 1 of 2

Next: a real memory architecture

Today the agent remembers the conversation. Next, it remembers the user and the institution — preferences, patterns and past briefings, recalled semantically.

Now

Session memory

Per-conversation context — the recent turns replayed each round, so follow-ups and refinements work naturally within a session.

Next

Persistent user memory

Durable per-user memories in Oracle: preferences (“always EBC format”), defaults, corrections — carried across sessions, editable and auditable.

Then

Semantic memory

Embeddings + k-NN over OpenSearch using OCI GenAI embeddings: the agent recalls similar past briefings, and topic/presenter matching becomes semantic instead of literal.

  • Episodic recall — “what did we do for their last visit?” answered from memory, not manual digging.
  • Memory-aware ranking — presenter and agenda suggestions weighted by what actually worked before.
  • RBAC applies to memory too — recalled content passes the same access gate as live data.
  • All Oracle-scoped — memories in Autonomous DB, vectors in OCI OpenSearch, embeddings from OCI GenAI. Nothing leaves the VCN.

15Roadmap · 2 of 2

Next: full-blown RBAC

The row-level gate on events is live. The roadmap deepens the same architecture — same resolver, same chokepoint — into finer scopes.

Enforced today

  • Row-level scope on events — role-compiled filter AND-ed into every read.
  • Activity scoping via the event join — pinned-event checks, capped joins, overflow denies.
  • Fail-closed resolution + index allowlist + banned-key guard.
  • Write-side — platform token per user on every mutation.

Planned rungs

  • Location & entity scoping — center-level and org-level visibility boundaries.
  • Private records — creator-only briefings invisible to everyone else.
  • Field-level redaction — sensitive values (opportunity size, exec attendees) masked by role, not just rows.
  • A write gate — policy checks on push actions in the agent itself, before the platform is even called.
  • Access audit trail — every scoped read logged: who asked, what filter applied, what returned.
