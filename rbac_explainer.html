🔐 Role-Based Access — How it fits our architecture

Calendar Insights · the LLM writes the query, but the server decides what data it's allowed to touch.

1 · How it works today NO ACCESS CONTROL

The LLM writes an OpenSearch query and it runs exactly as written. Nothing scopes it to a customer or a person.

The hole: customer_id is read from the header (api.py:164) but never used to filter. If the LLM forgets a filter — or is tricked by "show me all customers" — the data comes back. Everyone effectively sees everything.

2 · Proposed: enforce at ONE chokepoint SERVER-ENFORCED

We never ask the LLM to behave. We let it write whatever it wants, then wrap its query in a filter it can't escape — at the single function every query flows through.

3 · What "wrapping" actually means

The LLM's query gets demoted into a sub-clause. The green lines are what the server adds — the LLM never sees them and can't remove them. OpenSearch ANDs them together, so the LLM can only ever narrow within the allowed box.

{ "query": {
    "bool": {
      // ▼▼▼ server injects this — the user can't touch it ▼▼▼
      "filter": [
        { "term": { "...customerName.keyword": "ACME" } },   // tenant — EVERYONE gets this
        { "bool": { "should": [                              // ownership — only for sales_rep
          { "term": { "...oracleHostEmail.keyword": "me@acme.com" } },
          { "term": { "...requesterEmail.keyword":  "me@acme.com" } }
        ] } }
      ],
      // ▼ whatever the LLM wrote — now just a sub-clause ▼
      "must": [ { /* LLM's original query: "events about cloud migration" */ } ]
    }
} }
Key idea: we are not adding a "role" field to the data. The role is just an if-statement on the server that decides which filter conditions to inject. The conditions themselves are matched against fields that already exist (customer, host email, etc).

4 · Two layers of scope (they stack)

🅰️ Layer A — Tenant isolation

"You only see your own company." Applies to everyone, admin included. This is the urgent gap today. Ship first.

Filter: customerName = ACME

🅱️ Layer B — Role / ownership

Within a company, roles see different rows. Built on top of Layer A.

Extra filter: host/requester = me (only for restricted roles)

RoleInjected filterSees
admin customerName = ACME Everything for their company
sales_rep customerName = ACME AND (host = me OR requester = me) Only events they own / requested / present

6 · How I'd build it

1
Trust the role header

Briefing app injects x-cloud-user-role like the other headers. (Add JWT verification only if the endpoint is reachable directly & spoofable.)

2
Resolve an access_context at request entry

In api.py, alongside user_info, build {role, email, customer} → thread it down like user_info already flows.

3
Add a policy map: role → filter clauses

One small function. Admin → tenant only. sales_rep → tenant + ownership.

4
Inject in search() / count()

The single chokepoint. Wrap the LLM body in the bool/filter. Tenant filter goes in regardless of role.

5
(Later) field-level redaction

Strip sensitive fields (revenue, opportunity) from _source per role.

Additive & centralized — one inject function + one policy map + threading a context object. No rewrite of the LLM/tool layer.
